## Phase 98: FP8 Foundation

**Goal:** Add FP8 type support and CUDA kernels for E4M3/E5M2 formats

**Requirements:**
- FP8-01: FP8 type definitions (E4M3, E5M2) with conversion to/from FP32
- FP8-02: CUDA kernel for FP8 matrix multiplication (with FP32 accumulation)
- FP8-03: FP8 activation functions (ReLU, GELU) with numerical safety

**Success Criteria:**
1. E4M3 and E5M2 type definitions with IEEE 754-like semantics
2. Conversion kernels: FP32 → FP8, FP8 → FP32
3. FP8 GEMM kernel with FP32 accumulation buffer
4. Clamp behavior for overflow (max/min values)
5. GELU and ReLU activations in FP8 with safe downcast
6. Unit tests: 100% coverage for type conversions
7. Numerical accuracy: <0.1% relative error vs FP32 reference

**Implementation Notes:**
- Use struct with uint8_t storage, operator overloads for conversion
- FP8-E4M3: 4-bit exponent, 3-bit mantissa, range ~240 values
- FP8-E5M2: 5-bit exponent, 2-bit mantissa, range ~240 values
- Graceful fallback to FP16 if CUDA 20 FP8 intrinsics unavailable
- Follow quantize_tensor.hpp patterns for API consistency

## Phase 99: CUDA Quantization Kernels

**Goal:** Migrate CPU quantization to GPU with optimized CUDA kernels

**Requirements:**
- KERN-01: CUDA kernels for INT8/FP16 quantization (move from CPU)
- KERN-02: Vectorized load/store for quantization (128-bit wide)

**Success Criteria:**
1. cuda_quantize_f32_to_f16 kernel with vectorized load
2. cuda_quantize_f32_to_int8 kernel with asymmetric scaling
3. cuda_dequantize_int8_to_f32 kernel with scale broadcast
4. 128-bit vectorized I/O (float4, int4) for memory coalescing
5. Shared memory tiling for calibration histograms
6. Async copy overlap with compute streams
7. Unit tests: Correctness vs CPU reference implementation
8. Performance: >10x speedup over CPU quantization for 1GB tensors

**Implementation Notes:**
- Extend existing quantize_tensor.hpp with CUDA versions
- Use cuda::memory::Buffer<T> for device memory
- Implement as free functions in nova::quantize::cuda namespace
- Follow linalg kernel patterns for thread/block sizing

---

## Phase 100: Calibration Infrastructure

**Goal:** Production-grade calibration utilities with histogram and percentile methods

**Requirements:**
- CAL-01: Histogram-based calibration with percentile selection
- CAL-02: MinMax and MSE calibration methods
- CAL-03: Per-channel calibration with channel-wise scales

**Success Criteria:**
1. HistogramCalibrator class with configurable bin count
2. Percentile-based threshold selection (default: 99.99%)
3. MinMaxCalibrator with tensor-wise scale computation
4. MSECalibrator optimizing scale for minimum squared error
5. PerChannelCalibrator with axis-0 scale reduction
6. Calibrator fusion: compute all methods in single pass
7. Calibration cache save/load (JSON format)
8. Unit tests for each calibration method
9. E2E test: Calibrated INT8 model within 0.5% accuracy of FP32

**Implementation Notes:**
- Calibration data: std::vector<float> histogram
- Thread-safe histogram accumulation via atomicAdd
- Cache format: JSON with scales, zero_points, metadata
- Follow checkpoint module patterns for cache serialization
- Performance: <10s for 100GB dataset sample calibration

---

## Phase 101: QAT & Mixed Precision

**Goal:** Quantization-aware training patterns and automatic mixed precision management

**Requirements:**
- QAT-01: FakeQuantize op pattern for training integration
- QAT-02: Straight-through estimator (STE) gradient approximation
- MIX-01: Automatic mixed precision (AMP) manager with precision selection
- MIX-02: Layer-wise precision assignment based on sensitivity analysis

**Success Criteria:**
1. FakeQuantize forward: quantize → dequantize in FP32
2. FakeQuantize backward: STE pass-through gradient
3. QAT module with observer hooks for activation ranges
4. AMPManager class with precision policy configuration
5. SensitivityAnalyzer: measure per-layer loss gradient magnitude
6. Auto-precision assignment: FP8 for robust ops, FP16 for sensitive
7. Mixed precision matmul dispatch based on layer config
8. Unit tests: QAT forward/backward correctness
9. Integration test: Train with QAT, evaluate quantized model

**Implementation Notes:**
- QAT: Fake quantize with straight-through gradient (identity backward)
- AMP: Track precision per-layer in config struct
- Sensitivity: Run forward/backward, measure ||grad|| per layer
- Use existing mixed_precision from quantize_ops.hpp as base

## Phase 102: Benchmark & Integration

**Goal:** Comprehensive benchmark suite and final integration validation

**Requirements:**
- BENCH-01: Quantization benchmark suite (FP32 vs FP16 vs INT8 vs FP8)
- BENCH-02: Accuracy comparison tool with reference baseline

**Success Criteria:**
1. Benchmark harness with warmup, runs, and statistical reporting
2. Benchmark categories: matmul, activation, calibration, E2E
3. Throughput metrics: TFLOPS, GB/s for each precision
4. Latency metrics: p50, p95, p99 per kernel
5. Accuracy comparison: KL divergence, relative L2 error
6. Memory footprint comparison across precisions
7. Integration with v2.11 performance tooling (NVTX annotations)
8. Dashboard export: Precision comparison charts
9. Documentation: QUANTIZATION.md with usage examples
10. CI integration: GitHub Actions runs benchmark suite

**Implementation Notes:**
- Reuse benchmark infrastructure from v1.7 and v2.11
- Use PerformanceMetrics from v2.11 for instrumentation
- Dashboard: Extend v2.11 dashboard with precision comparison
- Benchmark workloads: Transformer attention, MLPs, convolutions
- Accuracy datasets: Synthetic + real weight distributions
